<p class="lede not-content">
  <strong>nidus (Latin, "nest") is a vector store, not a database.</strong> Chunk
  some source → embed each chunk → store the vectors and metadata → ask for
  nearest neighbours. It does exactly that and nothing more, so it{" "}
  <strong>compiles in seconds</strong> and embeds as a normal Rust dependency —
  no C++ toolchain, no SQL engine, no ten-minute cold build.
</p>

<p class="section-kicker not-content">Why it exists</p>

<div class="feature-grid not-content">
  <article class="feature">
    <span class="num">01</span>
    <h3>Exact or approximate search</h3>
    <p>Exact by default — 100% recall, fast at the target scale (≤ a few million vectors, comfortably in RAM). Score by cosine, dot, or Euclidean. Opt into an approximate index (HNSW or IVF) or int8 quantization when you want to trade some recall for speed at larger scale.</p>
  </article>
  <article class="feature">
    <span class="num">02</span>
    <h3>Pure-Rust, zero FFI</h3>
    <p>Never a crate that compiles C or links a native library. <code>#![forbid(unsafe_code)]</code>. cargo build is just rustc — and the whole crate runs under Miri, file IO included.</p>
  </article>
  <article class="feature">
    <span class="num">03</span>
    <h3>One directory, crash-safe</h3>
    <p>An append-only flat-f32 matrix plus a framed, CRC-checked op log. A crash loses at most the in-flight batch; a torn tail is recovered on open.</p>
  </article>
  <article class="feature">
    <span class="num">04</span>
    <h3>Typed metadata &amp; filters</h3>
    <p>Attach Str / Int / Bool / List / Null attributes and filter — equality, ranges, globs, sets, and negation — before scoring. Many collections share one embedding space, searchable in one call.</p>
  </article>
</div>
